Security Justification & Data Governance

This document outlines the purpose, necessity, and security posture of each API permission configured for the application. It is intended to provide full transparency regarding how these permissions are used, what safeguards are in place, and why each permission is the minimum required to deliver the service.

## 1\. Our Core Security Principles

| **Guiding Philosophy: Zero Trust & Least Privilege**<br/><br/>Every permission configured in this application follows the principle of least privilege — we request only what is strictly necessary, only when it is needed, and only on behalf of the authenticated user. |
| --- |

The following principles govern all data access within our platform:

*   User Impersonation Only: All permissions are of Delegated type, meaning the application acts on behalf of the signed-in user — never autonomously or with its own elevated identity.
*   No Background Operations: The system performs no scheduled jobs, background syncs, or silent data collection. Every operation is triggered explicitly by a user action.
*   Bounded by User’s Own Permissions: If a user does not have access to a resource in the Azure environment, our app cannot access it either. We cannot exceed the user’s native permissions.
*   No Data Retention of Sensitive Assets: We do not store, cache, or replicate raw data from your Azure environment. Metadata retrieved for display or query purposes is transient.
*   Consent-Driven: All access requires the user’s active session and consent. No permission allows us to operate without a user’s explicit request.

## 2\. Permission-by-Permission Justification

## 2.1 Azure Storage — user\_impersonation

| **Permission Scope** | **Type** | **Purpose & Justification** | **What We Do NOT Do** |
| --- | --- | --- | --- |
| user\_impersonation | Delegated | Used to read output artifacts stored in Azure Blob Storage — specifically the response payloads generated by Python code executed within the client’s own environment in response to a user query. This is a read operation on designated output containers only. | No write, delete, or overwrite of storage data. No access to arbitrary storage accounts or containers beyond designated output paths. |

| **Why Azure Storage access is needed**<br/><br/>When a user submits a natural language query that triggers Python execution in their environment, the result is written to a pre-configured Azure Blob Storage location within the client’s own tenant. Our application reads from that location to retrieve and display the response. This is a targeted, scoped read — not general storage browsing. |

## 2.2 Microsoft Graph — User.Read

| **Permission Scope** | **Type** | **Purpose & Justification** | **What We Do NOT Do** |
| --- | --- | --- | --- |
| User.Read | Delegated | Reads the authenticated user’s basic profile (name, email, user ID, tenant ID). Used for identity resolution, display personalization, and associating user actions with the correct account during the session. | No access to other users’ profiles, group memberships, mailboxes, calendars, or directory data. |

## 2.3 Power BI Service — Delegated Permissions

The Power BI Service permissions are the backbone of our platform’s core functionality. All permissions are Delegated, meaning they operate strictly within the signed-in user’s own access rights — the application can never access more than the user themselves is permitted to see.

These permissions collectively enable four key functional areas of the platform:

*   Lakehouse & Semantic Model Discovery: We read workspace contents, dataset definitions, and Lakehouse connection metadata so users can explore and query their data assets. This includes understanding which datasets are backed by Azure Data Lake or Synapse, and surfacing the right schema information for user queries.
*   Workspace & RBAC Awareness: We read workspace membership and role assignments to accurately reflect what the user has access to, and to respect permission boundaries within their organisation. We never modify workspace membership or user roles.
*   On-Demand Query Execution: When a user submits a query, we execute it against the relevant semantic model or dataset on their behalf using their own credentials. All executions are user-triggered — there are no scheduled or background operations.
*   Capacity & App Metadata: We read capacity and app information to understand the environment’s configuration and present the correct options to the user. No capacity changes or app modifications are made without explicit user instruction.

| **Why are there so many permission scopes?**<br/><br/>The Power BI REST API is highly granular — Microsoft divides functionality into many fine-grained scopes (e.g. separate scopes for reading reports vs. dashboards vs. datasets vs. workspaces). The number of scopes reflects the breadth of Power BI features our platform supports, not the breadth of data we access. Each scope is the minimum required for a specific feature, and all are Delegated — always bounded by what the signed-in user can access. |

## 3\. What We Explicitly Do Not Request

As important as what we request is what we deliberately choose not to request. The following categories of permissions are not configured and are not used:

| **Permission Category** | **Why We Do Not Request This** |
| --- | --- |
| **Application Permissions (non-delegated)** | All permissions are Delegated. Application permissions would allow server-to-server access independent of user sessions — we do not use these anywhere in the platform. |
| **User directory enumeration (User.ReadBasic.All, Directory.Read.All)** | We only need the signed-in user’s identity. We have no need to enumerate other users, groups, or directory objects. |
| **Mail, Calendar, Teams, SharePoint** | The platform has no communication or collaboration features. These APIs are entirely outside our scope. |
| **Key Vault direct secret access** | We do not read, modify, or manage secrets, keys, or certificates in Key Vault directly. |
| **Storage write or delete** | All Storage access is read-only for designated output containers. |
| **SQL DML / DDL execution** | We do not issue data manipulation or schema modification statements against any SQL endpoint. |
| **Power BI admin APIs** | We use standard user-context APIs only. Admin-level APIs that bypass workspace RBAC are not requested. |

## 4\. How User Impersonation Protects the Client

All permissions in this configuration are of type Delegated with user\_impersonation or user-context scopes. This is a critical design choice with significant security implications:

| **What ‘Delegated’ means in practice**<br/><br/>A Delegated permission means the application presents the user’s own identity token to the downstream Azure service. The Azure service evaluates the request as if the user themselves made it — applying all of the same RBAC, row-level security, and access controls that would apply to that user directly. |
| --- |

*   If the user does not have access to a specific workspace, dataset, or storage container, our application cannot access it either.
*   The application cannot access resources when the user is not logged in. There is no persistent background token or service principal operating independently.
*   Every access event is attributable to an authenticated user identity and will appear in your Azure audit logs accordingly.
*   Revoking a user’s Azure AD access immediately revokes the application’s ability to act on their behalf.

## 5\. Audit, Transparency & Client Controls

Clients retain full visibility and control over all access made through these permissions:

*   Azure AD Sign-in Logs: All delegated access events are recorded in your Azure Active Directory sign-in logs, attributable to the user’s identity.
*   Power BI Activity Log: All Power BI API calls (dataset queries, workspace reads, etc.) are logged in the Power BI activity log available to tenant admins.
*   Conditional Access Policies: Your organization’s existing Conditional Access policies apply to our application. MFA requirements, device compliance, and IP restrictions are all honoured.
*   Revocation: Admin consent can be revoked at any time from the Azure AD Enterprise Applications portal, immediately terminating all access.
*   Scope Review: Clients may request a full review of API call logs or a detailed technical walkthrough of how each permission is exercised at any time.

We do not collect, store, or transmit raw data from your environment beyond what is required to render a response to an active user request. |
