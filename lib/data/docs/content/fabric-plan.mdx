## Components

## Backend

*   Backend is written in Python language using FastAPI framework and containerized using Docker.
*   All endpoints are end-to-end encrypted using AES-128-CBC algorithm.
*   It generates a unique session ID for each user and stores it in the session store.

## UI

*   UI is written in React.js and containerized using Docker.
*   All requests are end-to-end encrypted using AES-128-CBC algorithm.

## Microsoft Fabric

*   All the user data resides on Microsoft Fabric only.
*   No data can be read or written by Backend without authentication.

## Qdrant

*   Qdrant is used for vector storage and retrieval.
*   Only the structure of the data is stored in Qdrant, not the actual data.

## Azure Foundry

*   Different sets of LLMs are used according to the difficulty of the problem.
*   All the LLMs are deployed on Azure Foundry.
*   Only the structure of the data is sent to LLMs for code generation, not the actual data.

## PostgreSQL

*   This database is used to store the usage metrics of each user and Tenant Subscription Information.
*   Credits are updated at both the start and end of the user request.

## ELK Stack

*   This stack comprises Elasticsearch, Logstash, Kibana, and Beats.
*   It is used to store user feedback.

## Analytics and Visualization

*   All the data is read using Pandas DataFrames and visualized using Plotly and AM Charts.
*   The code execution happens at the OneLake of client, and the results are sent to the backend to reformat and render on UI.

## **Application Flow**

![](/images/extracted/img_57d2ec9c.png)

## Authentication and Session Initialization

1.  The user goes through the Microsoft Authentication layer through the UI.
2.  Upon success, a unique user token is generated, and Microsoft Fabric renders Karl – Insights Agent UI as iframe with this user token. If authentication is unsuccessful, Microsoft shall redirect the user to Microsoft login page.
3.  The user token is stored in the iframe session and used for the processes to work as authentication is mandated and required by Microsoft.
4.  Backend decrypts the user's token, generates a process bound OBO token and gets all the warehouses and lakehouses of the user ‘s workspace from Fabric.
5.  Backend responds with the fetched list to the UI.
6.  The user can select any warehouse or Lakehouse from the list, and the selection is sent to the Backend.

## Data Extraction Phase

1.  The user can ask to fetch their data or a subset of their data in the UI. The query is sent to the Backend.
2.  Backend checks whether the user has enough credits to run the query. This is done by checking the usage metrics in the PostgreSQL database. If the user has enough credits, the flow goes to step 9. If not, a low-balance response is sent to the UI to inform the user.
3.  Backend fetches relevant SQL and DDL statements along with the documentation, if available, from the Qdrant vector database. At any point, the user can see and edit the data present in the database from the UI.
4.  Backend then sends the user's query along with the fetched relevant data to an LLM deployed on Azure Foundry. Here, the LLM gets only the structure of the data, not the data itself.
5.  The LLM generates SQL code corresponding to the user request.
6.  Backend performs code sanitization and checks, such as ensuring the SQL command should only read the data, not write or delete. Then, the SQL command is executed on the selected warehouse or Lakehouse in Fabric.
7.  Fabric sends the data to the Backend upon successful execution, and the Backend saves it as either a CSV or a Parquet file.
8.  Backend sends the summary of the data to the same LLM.
9.  The LLM outputs a natural language response based on the data summary to the Backend.
10.  Backend clubs with the data and the LLM response jand send them to the UI.
11.  Then, Backend updates the usage metrics in the PostgreSQL database.
12.  The user can like or dislike the output from the UI using the thumbs-up or thumbs-down buttons, respectively, and the feedback is sent to the Backend.
13.  Backend stores user feedback in the ELK Stack.

Note:

*   Steps 7 to 19 (Data Extraction Phase) will repeat for each user request. Unique subsets of data and unique responses are shown to the user for every unique query.
*   At any point, if some error occurs, instead of sending the error to the UI directly, a unique natural language response generated by another LLM is sent to the UI.

## Data Visualization Phase

1.  If the user is satisfied with the last fetched data, they can tap the Start Visualization button in the UI to fix the data and start asking questions and get visualizations.
2.  Backend again checks the user's credits.
3.  Backend sends the user query along with the summary of the data to an LLM.
4.  With the help of tool calling, the LLM chooses whether to perform deep analysis or not. If deep analysis is not required, it directly gives Python code (step 26).
5.  If deep analysis is required, the LLM sends its thoughts to the Backend.
6.  For every thought, Backend sends it along with the data summary to another LLM.
7.  The LLM generates some Python code according to the user's needs.
8.  Backend performs the code sanitization again and executes the code in a controlled and restricted environment with memory protection, CPU limits, timeouts, etc.
9.  Upon successful execution, the code outputs a Pandas DataFrame and a Plotly chart.
10.  The summary of the results is then passed to the same LLM again, which has the context of the session.
11.  The LLM outputs a natural language response for the results.
12.  Backend then clubs the results like charts, code, etc., with the LLM response and sends them to the UI. The UI then uses AM Charts to display the Plotly chart with more interactive elements and a variety of charts with similar functionality.
13.  Backend then updates the user's credits in the PostgreSQL database.
14.  The user can like or dislike the output from the UI using the thumbs-up or thumbs-down buttons, respectively, and the feedback is sent to the Backend.
15.  Backend stores user feedback in the ELK Stack.

Note:

*   Steps 20 to 34 (Data Visualization Phase) will repeat for each user request. Unique subsets of data, unique responses, and unique charts are shown to the user for every unique query.
*   At any point, if some error occurs, instead of sending the error to the UI directly, a unique natural language response generated by another LLM is sent to the UI.
*   For deep analysis, we follow the Tree of Thoughts (ToT) approach.

## **Infrastructure Architecture**

![](/images/extracted/img_bcf1fe31.png)

The architecture is composed of three main logical components:

1.  **Frontend:** Developed using React and the Fabric Workload Development Kit, this component provides the user interface for the application.
2.  **Backend:** Built with C# and the Fabric Workload Development Kit, the backend handles core application logic, data processing, and interactions with other services.
3.  **Python Microservice:** A Dockerized microservice dedicated to AI and machine learning workloads and complex computations. This service is critical for processing data and generating insights.

These components operate as a custom workload within the Microsoft Fabric environment, leveraging the capabilities of the Fabric Workload Development Kit for seamless integration. The entire application is hosted on an Azure Virtual Machine (Azure VM), forming part of the broader Azure Cloud Infrastructure. Data persistence and storage are managed through OneLake Lakehouse/Warehouse, ensuring robust data management within the Microsoft Fabric platform.

## Security Documents

What are security measures have been covered and gaps.

1.  The Data being Stored on Kanerika server
    1.  Resolution – Now directing the data saving and code executions to user’s OneLake for processing.
2.  The user authentication –
    1.  Using Microsoft approved and suggested mechanism to acquire and use the user token.
    2.  Generating only the required OBO tokens and not any unused ones.
3.  API Communication –
    1.  Both to and from of API payload communication has been encrypted with end-to-end encryption using AES-128-CBC algorithm

### Security Skeleton

#### Identity & Access

*   Entra ID authentication
*   User → Group → Role mapping
*   Fabric workspace roles

#### API Security

*   OAuth2 / JWT validation
*   Service-to-service auth (managed identity)

#### Data Security

*   No data is flown outside Fabric
*   No raw credentials in code
*   All the generated code is being executed on client OneLake utilizing client Fabric Capacty.
*   All the code executions are logged in the monitoring hub of fabric.

#### AI Safety

*   Prompt logging
*   Input/output moderation
*   No training on customer data
*   No factual data of customer is pushed to LLM only the essence / metadata for reliable code generation and responses.

### Architecture level services

*   UI Service
*   Backend Service
*   Vector DB Microservice
*   ELK Service

### Approvals & Governance Flow

*   Code level
    1.  Developer pushed code will go through PR for Dev Approval
    2.  Managerial review prior to code commits to UAT Repo
    3.  Security review (API + data)
*   Then QAT – Automated and manual testing catered to features that have been updated or changed along with all existing functionality end to end testing.
*   Prod Deployment will commence post QAT approval.
